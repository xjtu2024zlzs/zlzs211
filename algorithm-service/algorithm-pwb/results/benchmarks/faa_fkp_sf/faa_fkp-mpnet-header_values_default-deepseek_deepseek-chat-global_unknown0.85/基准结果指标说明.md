# FAA FKP 全局 Top-K 基准结果 — 指标说明与分析

本文档对应同目录下的汇总 CSV：

`faa_fkp-mpnet-header_values_default-deepseek_deepseek-chat-global_unknown.csv`

说明**每一列指标的含义、计算方式**，并解释**实验文件夹命名中的参数**；便于读表与复现实验。

---

## 1. 实验场景（这份 CSV 是怎么跑出来的）

- **脚本**：`experiments/benchmarks/faa_benchmark_sf.py`，评估模式为 **`global_unknown`（全局 Top-K）**。
- **数据**：`data/magneto_faa_fkp`（源表 37 张，目标表多张；Ground Truth 来自 `ground-truth/_all_mappings.csv`）。
- **任务设定**：对每个**源表**的每一**源列**，从**所有目标表**的列中检索 Top-K（嵌入相似度），再经各方法（Magneto / Boost / LLM 等）得到最终匹配集合；评估时把**所有带 GT 的表对**上产生的匹配**合并**成一个大的 `MatcherResults`，与合并后的 GT 列对列表一起算指标。

因此：`total_gt_matches = 200` 表示合并评估的 **GT 列对总数**（不是表对数）。

## 2. 文件名与路径里的参数（要不要解释？）

**建议解释**：路径中的片段对应启动基准时的关键超参，与可复现性直接相关。

| 片段 | 含义 |
|------|------|
| `faa_fkp_sf` | 结果根目录：在 `faa_benchmark_sf.py` 中 `use_sf` 为真时 `benchmark` 名为 `{dataset}_sf`；本实验数据集为 `faa_fkp`。 |
| `faa_fkp` | 数据集名，对应 `--root` 默认 `data/magneto_faa_fkp` 等。 |
| `mpnet` | 嵌入模型键：`sentence-transformers/all-mpnet-base-v2`（见脚本中 `model_map`）。 |
| `header_values_default` | Magneto **列编码模式**（`--mode`）：表头 + 默认值等序列化方式。 |
| `deepseek_deepseek-chat` | **LLM 重排**使用的模型标识（如 `deepseek/deepseek-chat`），用于 MagnetoGPT / neo4j2 等需 LLM 的方法；无 LLM 的方法仍会跑，但配置里会带该实验名。 |
| `global_unknown` | **评估模式**：全局 Unknown Table，即每源列候选来自全库目标列 Top-K，而非单表对。 |

CSV 内的 `benchmark`、`dataset`、`method` 列与上述一致；**`method`** 为算法变体名称（如 `Magneto`、`MagnetoBoost`、`MagnetoGPT`、`MagnetoGPT_neo4j2`）。

## 3. CSV 各列含义与计算方式

### 3.1 元信息列

| 列名 | 含义 | 如何得到 |
|------|------|----------|
| **benchmark** | 基准任务命名空间 | 代码里多为 `faa_fkp_sf`。 |
| **dataset** | 数据集短名 | 如 `faa_fkp`。 |
| **method** | 匹配方法 | 一次实验一行：Magneto / MagnetoBoost / MagnetoGPT / MagnetoGPT_neo4j2 等。 |

| **gt_table_pairs** | 含 Ground Truth 的 (源表, 目标表) 对数 | 在 GT 字典中出现的表对个数（本文件为 **42**）。 |
| **total_gt_matches** | GT 列对总数 | 所有上述表对上 GT 列对合并后的条数（本文件为 **200**）。 |
| **total_runtime_seconds** | 该方法本模式总耗时（秒） | 各源表一次 `compute_global_topk` + 匹配逻辑 wall-clock 累加。 |

---

### 3.2 MRR（`mrr`）

- **含义**：**按源列聚合的 Mean Reciprocal Rank**。对每个在预测结果中出现的**源列**，在其候选列表（按分数从高到低）里找**第一个**落在该源列 GT 目标列集合中的位置 \(r\)，贡献 \(\frac{1}{r}\)；再对所有**在预测里出现过且 GT 中有标注的源列**取平均。
- **代码位置**：`experiments/benchmarks/benchmark_utils.py` 中的 **`compute_mean_ranking_reciprocal_adjusted`**。
- **注意**：若某源列在 GT 中有标注但预测里完全没有该源列的候选，则该源列**不参与**分母（实现里 `total_queries` 只统计 `ordered_matches.keys()` 中有 GT 的列）。这与「对所有 GT 源列求 MRR」的严格定义略有不同，读数时建议结合导出的一对一/全量匹配 CSV 一起看。

---

### 3.3 `Recall@20`（列名与实现不一致，重要）

- **CSV 列名**：`Recall@20`
- **实际实现**：`benchmark_utils.py` 中的 **`calculate_recall_at_k`**，**没有使用数字 20**，也**不是**「每个源列只看 Top-20 候选」的经典 Recall@20。
- **实际算法**：
  1. 将 GT 转为集合（每条 GT 用无序列对 `frozenset({source_col, target_col})` 表示，避免左右顺序问题）。
  2. **按 `MatcherResults` 字典迭代顺序**遍历**每一条预测边** `((src_tbl, src_col), (tgt_tbl, tgt_col), score)`；若该列对命中某条尚未被计数的 GT，则命中数 +1 并从 GT 集合中移除该条（每条 GT 最多计 1 次）。
  3. **返回值** = 命中 GT 条数 / `len(ground_truth)`（即 **200** 的分母）。

因此，该列更准确的描述是：**在完整预测集合的某种遍历顺序下，最多能覆盖多少比例的 GT 列对（去重后）**。数值上常与「预测里是否包含正确列对」高度相关；**不要**把它理解成标准论文里按 query 截断 Top-20 的 Recall@20。

若需真正的 Recall@K，需要在代码里按源列截断 Top-K 后另行实现。

### 3.4 Valentine：`All_*` 一组（全量预测字典上的度量）

以下五项来自 **Valentine** 的 `MatcherResults.get_metrics(ground_truth)`，输入为**合并后的完整预测** `MatcherResults`（多对多、多候选均保留，只要仍在字典中）。

实现见（以本仓库为准）：`valentine` 包内 `MatcherResults` 与 `valentine.metrics`。

| 列名 | Valentine 键名 | 含义与计算要点 |
|------|----------------|----------------|
| **All_Precision** | `Precision` | **精确率** = TP / (TP + FP)。在默认 `METRICS_CORE` 下，**Precision / Recall / F1 / PrecisionTopNPercent** 会先对匹配结果做 **`one_to_one()`**（按分数贪心的一对一子集），再数 TP/FP/FN（GT 为 `(源列名, 目标列名)` 二元组列表）。 |
| **All_F1Score** | `F1Score` | **F1** = \(2PR/(P+R)\)，基于上式同一套 TP/FP/FN。 |
| **All_Recall** | `Recall` | **召回率** = TP / (TP + FN)。 |
| **All_PrecisionTop10Percent** | `PrecisionTop10Percent` | 先 **`one_to_one()`**，再取预测中 **分数最高的 10%** 条（`take_top_percent(10)`），在这子集上算 Precision（TP/(TP+FP)）。 |
| **All_RecallAtSizeofGroundTruth** | `RecallAtSizeofGroundTruth` | **不**在度量内部再做 `one_to_one`：取预测按分数排序后的 **前 \|GT\| 条**（`take_top_n(len(ground_truth))`），在这 **200 条** 子集上算 Recall = TP/(TP+FN)。反映「若只允许输出与 GT 同数量的预测，能召回多少 GT」。 |

**与 `One2One_*` 的关系（代码层面）**：

- 脚本中先 `all_metrics = matches.get_metrics(gt)`，再 `one2one = matches.one_to_one(); one2one_metrics = one2one.get_metrics(gt)`。
- Valentine 默认 **`Precision` / `Recall` / `F1Score` / `PrecisionTopNPercent`** 的 `one_to_one=True`，即内部会先把输入转成一对一子集。对**已是** `one_to_one()` 的结果再 `get_metrics`，理论上 **P/R/F1/PrecisionTop10%** 与在全量上 `get_metrics`（内部再 one_to_one）**应相同**。
- **`RecallAtSizeofGroundTruth`** 不使用 `one_to_one`，直接对传入的 `MatcherResults` 做 `take_top_n(|GT|)`，因此 **All_RecallAtSizeofGroundTruth** 与 **One2One_RecallAtSizeofGroundTruth** 通常**不同**（全量排序截断 vs 一对一子集排序截断）。

若你观察到 **All_Precision 与 One2One_Precision**（或 R、F1）数值差异很大，应核对当时安装的 **`valentine` 版本**及 `METRICS_CORE` 是否被改动；在官方默认配置下这几项应对齐。

---

### 3.5 Valentine：`One2One_*` 一组（先 one_to_one 再度量）

| 列名 | 含义 |
|------|------|
| **One2One_Precision** | 对 **`matches.one_to_one()`** 的结果再 `get_metrics` 得到的 Precision（内部仍可能再 one_to_one，一般等价于上面分析）。 |
| **One2One_F1Score** | 同上，F1。 |
| **One2One_Recall** | 同上，Recall。 |
| **One2One_PrecisionTop10Percent** | 一对一子集上的 Top10% Precision。 |
| **One2One_RecallAtSizeofGroundTruth** | 一对一子集上取 top-|GT| 后的 Recall（见上）。 |

**一对一子集 `one_to_one()`**（Valentine）：在按分数排序的匹配上贪心保留边，使每个源列、每个目标列至多出现一次（具体实现见 `valentine.algorithms.matcher_results.MatcherResults.one_to_one`）。

## 4. 本文件四行结果速读（便于对比方法）

| method | mrr | 备注 |
|--------|-----|------|
| Magneto | ~0.847 | 纯嵌入全局 Top-K + 重排，最快。 |
| MagnetoBoost | ~0.842 | 约束增强（如 pgvector/SF 路径），略快于 LLM。 |
| MagnetoGPT | ~0.912 | LLM 重排，MRR 最高之一，耗时约 3500s 量级。 |
| MagnetoGPT_neo4j2 | ~0.909 | 两轮 LLM + Neo4j 语义边，与 GPT 接近，耗时类似。 |

同一文件中 **`Recall@20` 列四行均为 0.86**：由 `calculate_recall_at_k` 定义决定，在该次合并预测下四种方法覆盖了相同比例的 GT（或遍历命中顺序导致相同计数），**不**代表四种方法在细粒度列级表现完全一致。

## 6. 是否需要解释「每一个参数」？

- **CSV 列**：建议**全部**按上表理解，写论文或报告时应对 **MRR、`Recall@20` 真实实现、RecallAtSizeofGroundTruth** 单独说明，避免与文献中同名指标混淆。
- **路径/实验名中的片段**：与**可复现性**相关，建议在附录或实验设置中**逐项对应**命令行（`--embedding_model`、`--mode`、`--llm_model`、`--eval_mode` 等）。
- **算法内部超参**（如 Top-K 具体取值、温度、prompt）：不在此 CSV 中；需查阅 `GlobalMatcher`、`llm_reranker` 及当时运行日志或脚本默认值。
