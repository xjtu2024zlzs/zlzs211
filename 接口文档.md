# 本微服务与其他微服务数据交互接口文档

## 1. 文档说明

本文档依据《系统数据输入与输出.txt》中的数据交互要求，并以本系统 `project3` 模块现有数据结构和接口实现为准，设计本微服务与课题一、课题二、课题五等外部微服务之间的数据输入与输出接口。

本系统服务标识：`project3`

默认网关前缀：按部署环境配置，例如 `/project3` 或经网关转发后的根路径。下文接口路径均为业务服务内部路径。

统一响应格式沿用若依 `AjaxResult`：

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": {}
}
```

任务状态枚举：

| 状态 | 含义 |
| --- | --- |
| `PENDING` | 已提交，等待执行 |
| `RUNNING` | 执行中 |
| `SUCCESS` | 执行成功 |
| `FAILED` | 执行失败 |
| `CANCELED` | 已取消 |

算法任务类型：

| task_type | 含义 | 主要输出去向 |
| --- | --- | --- |
| `KQC_MINING` | 关键质量特性挖掘 | 课题一、课题五 |
| `KEY_PROCESS` | 关键工序识别 | 课题一 |
| `PROCESS_ANOMALY_DETECT` | 工序异常检测 | 课题一 |
| `FEATURE_ANALYSIS` | 质量/状态数据特征分析 | 本系统内部链路 |
| `FEATURE_PROCESSING` | 特征处理 | 本系统内部链路 |
| `EARLY_DEGRADATION_POINT_DETECT` | 早期故障点/退化点检测 | 课题一、课题五 |
| `FAULT_PREDICT` | 故障时间预测 | 课题一、课题五 |
| `FRAME_BEAM_CRACK_IDENTIFY` | 框梁裂纹识别 | 本系统服务识别结果 |

## 2. 核心数据对象

### 2.1 质量对象层级

本系统质量对象按如下层级组织：

`aircraft` 飞机 -> `subsystem` 分系统 -> `equipment` 设备 -> `component` 组件 -> `part_template` 零件模板 -> `part_instance` 零件实例

树节点统一结构：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | string | 是 | 节点全局标识，格式如 `aircraft:A001`、`component:C001` |
| `parent_id` | string | 否 | 父节点标识 |
| `name` | string | 是 | 节点名称 |
| `node_type` | string | 是 | `aircraft/subsystem/equipment/component/part_template` |
| `level` | integer | 是 | 层级，飞机为 1，零件模板为 5 |
| `terminal` | boolean | 是 | 是否终端节点 |
| `child_count` | integer | 否 | 子节点数量 |
| `part_count` | integer | 否 | 下级零件实例数量 |
| `route_name` | string | 否 | 工艺路线名称 |
| `process_count` | integer | 否 | 工序数量 |

零件实例结构：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `part_instance_id` | string | 是 | 零件实例编号 |
| `part_template_id` | string | 是 | 零件模板编号 |
| `part_number` | string | 是 | 零件编号 |
| `part_name` | string | 是 | 零件名称 |
| `material` | string | 否 | 材料 |
| `specification` / `spec_model` | string | 否 | 规格型号 |
| `serial_number` | string | 否 | 序列号 |
| `batch_number` | string | 否 | 批次号 |
| `manufacturer` | string | 否 | 制造商 |
| `production_date` | string | 否 | 生产日期，建议 `yyyy-MM-dd` |
| `current_status` / `status` | string | 否 | 当前状态 |
| `quality_level` | string | 否 | 质量等级 |
| `key_degree` | string | 否 | 关键程度 |

### 2.2 算法任务结果

本系统所有对外算法结果统一沉淀到 `t3_algorithm_task_results`，接口返回字段如下：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `taskId` | string | 任务编号 |
| `taskType` | string | 任务类型 |
| `taskName` | string | 任务名称 |
| `status` | string | 任务状态 |
| `requestId` | string | 外部请求幂等编号 |
| `algVersion` | string | 算法版本 |
| `aircraftId` | string | 飞机编号 |
| `subsystemId` | string | 分系统编号 |
| `equipmentId` | string | 设备编号 |
| `componentId` | string | 组件编号 |
| `bizId` | string | 业务对象编号 |
| `bizLevel` | string | 业务对象层级 |
| `bizName` | string | 业务对象名称 |
| `featTaskId` | string | 关联特征分析任务编号 |
| `summary` | string | 摘要结果 |
| `resultVal` | string | 结果值 |
| `resultUnit` | string | 结果单位 |
| `reqJson` | string | 请求 JSON |
| `resJson` | string | 完整结果 JSON |
| `errMsg` | string | 错误信息 |
| `startAt` | datetime | 开始时间 |
| `endAt` | datetime | 完成时间 |

## 3. 输入接口

### 3.1 接收课题一质量对象数据

用于接收飞机、分系统、设备、组件、零件模板、零件实例等层级基础数据。

当前系统已有文件导入/API 导入入口：

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `POST` | `/monitor/text/hierarchy/import` | 上传层级对象 Excel 文件 |
| `POST` | `/monitor/text/api/import` | 从 API 请求体导入层级对象文件 |
| `GET` | `/monitor/text/hierarchy/template` | 下载层级对象导入模板 |

推荐外部微服务调用的 JSON 口径：

`POST /monitor/text/api/import`

```json
{
  "taskType": "hierarchy",
  "apiUrl": "http://topic-one-service/files/hierarchy_20260609.xlsx",
  "fileName": "hierarchy_20260609.xlsx"
}
```

### 3.3 接收状态监测数据（时序型）

用于接收振动、温度、压力、电流/功率、转速、载荷、运行过程时间序列等服役状态数据。

当前系统对时序数据主要以文件方式提交到故障识别、工序异常检测、框梁裂纹识别任务。

建议外部微服务先按本系统样本文件目录或样本记录导入，再启动分析任务。

    "id": "component:C001",
    "name": "液压泵组件",

## 6. 对接约束

1. 外部系统传入对象编号时，应优先使用本系统节点编号格式：`aircraft:{id}`、`subsystem:{id}`、`equipment:{id}`、`component:{id}`、`part_template:{id}`。
2. 算法类接口建议传入 `requestId`，用于外部请求追踪和幂等处理。
3. 大文件、时序数据、文本数据优先通过文件 URL 或分片上传方式传入，接口请求体只传文件元数据和算法参数。
4. 数值型质量数据字段可扩展，但必须保留 `part_instance_id` 或可追溯到本系统质量对象层级的业务编号。
5. 时间字段建议统一为 `yyyy-MM-dd HH:mm:ss`，日期字段建议统一为 `yyyy-MM-dd`。
6. 输出给课题一、课题五的数据以 `t3_algorithm_task_results` 中的任务结果为准，重点读取 `taskType`、`status`、`bizId`、`bizName`、`summary`、`resultVal`、`resultUnit`、`resJson`。
